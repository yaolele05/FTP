## 二.使用epoll的背景

### （1）旧模型（一连接一线程）

```
accept → thread → recv(block)
```

问题：

> 线程爆炸
>
> 内存爆炸
>
> CPU切换爆炸

------

# 阻塞模型撑不住高并发

例如：

```
10000个连接
```

如果：

```
一个连接一个线程
```

会出现：

```
10000个线程
```

而实际上：

```
99%的线程都在睡觉
```

因为：

```
大多数连接没数据
```

这会导致：

------

## 上下文切换爆炸

CPU不停：

```
线程A
→ 线程B
→ 线程C
```

切换寄存器。

成本非常高。



## 则使用epoll后就会

> 一个线程
>     ↓
> epoll_wait（等事件）
>     ↓
> 处理 ready fd

epoll的优点：

>不用轮询所有 socket
>
>内核帮你监视
>
>哪个 socket 可读/可写
>
>内核主动通知你

**epoll不是：**

 *多线程模型*
  *并发模型*



而是：

> IO事件通知机制

### （2）eventpoll

### ***Linux 内核创建了一个 eventpoll 对象***（三个核心）

内核里面：

```
struct eventpoll
{
    红黑树;
    ready链表;
    等待队列;
}
```

这才是真正的 epoll。

#### eventpoll 是“事件管理器”

它专门管理：

```
哪些socket被监听
哪些socket已经就绪
哪些线程正在等待事件
```

它内部最重要的：

#### 1.核心一：红黑树（RB Tree）

***原因：***epoll 需要：

```
快速：
ADD
DEL
MOD
查找fd
```

***复杂度：O(logN)***

`epoll_ctl(epfd, ADD, 5)
epoll_ctl(epfd, ADD, 8)
epoll_ctl(epfd, ADD, 100)`

作用：

“监控名单”

#### ***保存所有被监听的 fd***(注意不是保存哪些fd有数据)

#### 2.核心二：ready链表

例如：

```
fd=8 有数据可读
fd=100 可写
```

它们会进入：ready list

作用：

***epoll_wait 不扫描 fd而是“检查 ready list 是否为空”***

#### 3.核心三：wait queue ：callback 回调机制

#### epoll_wait()的模式是socket主动通知epoll

当 socket 收到数据：

会执行：

```
ep_poll_callback()
```

它会：

```
把fd加入ready list
```

于是：

```
epoll_wait()
```

立刻被唤醒，返回fd



### （3)eventpoll 和 epoll_event 结构体

`eventpoll` = 内核里的 epoll“实例/调度器”
 `epoll_event` = 用户态与内核交互的“事件描述单元”

# epoll_event：用户态“事件容器”

### 定义（用户态看到的）

```
struct epoll_event {
    uint32_t events;   // 事件类型（EPOLLIN/EPOLLOUT/...）
    epoll_data_t data;  // 用户自定义数据（fd / ptr）
};
```

# eventpoll：内核 epoll 核心

在 Linux 内核中：



```
struct eventpoll {
    struct rb_root rbr;        // 红黑树：管理所有监控 fd
    struct list_head rdllist;  // 就绪链表
    wait_queue_head_t wq;      // epoll_wait 阻塞队列

    struct list_head ovflist;  // 溢出链表（高负载用）

    struct file *file;         // epoll fd 对应的文件
};
```

| 对比项         | epoll_event           | eventpoll              |
| -------------- | --------------------- | ---------------------- |
| 所在层         | 用户态 + 内核交互接口 | 内核态                 |
| 本质           | 事件描述结构体        | epoll实例/调度器       |
| 是否存fd关系   | ❌ 不存                | ✅ 存（红黑树）         |
| 是否存就绪事件 | ❌ 不存                | ✅ 存（rdllist）        |
| 是否参与调度   | ❌ 不参与              | ✅ 核心调度器           |
| 生命周期       | 一次 epoll_wait       | 整个 epoll fd 生命周期 |

epoll_event 是用户态与内核之间传递事件的结构体，用于描述 fd 的事件类型和携带用户数据；
 eventpoll 是内核中的 epoll 实例结构，负责维护所有监控 fd 的红黑树、就绪队列以及 epoll_wait 的阻塞与唤醒机制，是 epoll 的核心调度器。

### （4）LT和ET模式：

### （1）什么是 LT

Level Trigger

条件触发。

只要：

```
recv queue 里还有数据
```

epoll_wait 每次都会返回。

即：

```
有数据
→ 一直通知
```

## （2）什么是 ET

Edge Trigger

边沿触发。

只有：

```
空 -> 非空
```

这一瞬间通知一次。

之后：

即使还有数据，
 也不会再通知。

#### 1. event loop（事件循环）

本质：

```
while(1)
{
    等待事件
    处理事件
}
```

一直循环。

因为服务器：

```
不能只处理一个客户端
```

而是：

```
必须永远运行
```

例如：

```
客户端A来了
客户端B来了
客户端C来了
```

服务器：

```
必须不断监听
不断处理
```

### 最原始的服务器循环

```
while(1)
{
    connfd = accept(...);

    recv(connfd,...);

    send(connfd,...);
}
```

问题：

```
一次只能服务一个客户端
```

**因为：recv阻塞了，无法accept其他客户端**

### 则有epoll IO多路复用

------

Reactor：

内部：

```
while(1)
{
    n = epoll_wait(...);

    for(...)
    {
        处理事件
    }
}
```


# 六、总结

## 1. epoll 的本质

epoll 不是并发模型，也不是线程模型，它本质是：

> **内核提供的“事件就绪通知机制”**

它做三件事：

```
1. 管 fd（红黑树）
2. 收集就绪 fd（ready list）
3. 通知用户态（epoll_wait）
```

epoll 不关心业务
 epoll 不关心线程
epoll 不关心 callback

------

## 2. Reactor 的本质（核心）

Reactor 不是 IO 机制，而是：

> **基于 epoll 的“事件驱动编排框架”**

它做的是：

```
epoll（拿事件）
   ↓
dispatch（分发事件）
   ↓
callback（执行逻辑）
```

所以 Reactor =

> **事件循环 + 分发器 + 回调系统**

------

## Reactor 高性能的核心原因

```
1. 不轮询 fd（epoll）
2. 不创建大量线程（单线程 event loop）
3. 不阻塞 IO（非阻塞 + 事件驱动）
```
